**버전**: 1.0

**발신 주소**: `noreply@updates.puzl.co.kr` (Resend API)

## 1. CRM 전략 개요

### 목적

Referio CRM의 핵심 목적은 **파트너의 첫 행동을 유도하고, 성과를 만들어내고, 수익을 확실히 전달하여 이탈을 방지하는 것**이다. 이메일은 단순 알림이 아니라 파트너가 다음 행동을 하게 만드는 "넛지 시스템"이다.

### 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **행동 유도 단일 초점** | 모든 이메일에 CTA 1개. 읽은 후 할 일이 명확해야 한다 |
| **돈의 흐름을 투명하게** | 커미션 확정, 입금 예정, 입금 완료 — 모든 금전 이벤트를 즉시 알린다 |
| **공감 + 실용** | "잘 하고 있다" 격려와 "구체적으로 이렇게 하라" 가이드를 함께 제공 |
| **중복 발송 절대 금지** | 발송 로그 테이블 기반 중복 방지. 파트너 신뢰는 한 번 잃으면 끝 |
| **개인화 우선** | 파트너명, 프로그램명, 커미션액 등 개인화 변수 적극 활용 |

### 파트너 라이프사이클 맵

```
가입 ──→ 온보딩 ──→ 활동 ──→ 성과 ──→ 정산 ──→ 리텐션
 │         │         │        │        │         │
 │    [이메일 1]  [이메일 4] [이메일 7] [이메일 10] [이메일 6]
 │    가입 축하   활동 넛지  첫 유입   정산 확정   월간 리포트
 │         │         │        │        │
 │    [이메일 2]  [이메일 5] [이메일 8] [이메일 11]
 │    참여 독려   신규 프로그램 첫 수익  정산 정보 요청
 │         │                  │        │
 │    [이메일 3]          [이메일 9] [이메일 12]
 │    승인 완료           티어 승급  입금 완료
 │
 └──→ 경계 상황
      [이메일 13] 가이드라인 위반 경고
      [이메일 14] 탈퇴 처리 완료
```

## 2. 이메일 시퀀스 전체 지도

| 3 | A-온보딩 | 파트너 승인 완료 + 활동 가이드 | 광고주 승인 클릭 | API hook (실시간) | 높음 (추천링크, 커미션, 가이드) | 즉시 구현 필수 |
| 4 | B-활동 | 활동 미감지 넛지 | 승인 후 7일 + 활동 0건 | Cron (일 1회) | 중 (프로그램별 추천링크) | 중요 |
| 5 | B-활동 | 신규 프로그램 안내 | 광고주 신규 프로그램 활성화 | API hook (이벤트) | 높음 (프로그램 상세, 커미션) | 선택 |
| 6 | B-활동 | 월간 성과 리포트 | 매월 1일 자동 집계 | Cron (월 1회) | 최고 (KPI, 증감율, 분기 콘텐츠) | 중요 |
| 7 | C-성과 | 첫 고객 유입 축하 | 첫 리드 유입 즉시 | API hook (실시간) | 중 (프로그램명, 유입 일시) | 즉시 구현 필수 |
| 8 | C-성과 | 첫 수익 확정 축하 | 첫 settlement 생성 | API hook (실시간) | 높음 (커미션액, 티어 진행률) | 즉시 구현 필수 |
| 9 | C-성과 | 티어 승급 축하 | 티어 승급 확정 | 배치 또는 실시간 | 높음 (티어 혜택, 진행률) | 선택 |
| 10 | D-정산 | 정산 확정 안내 | 광고주 정산 확정 (정보 등록 완료) | API hook (실시간) | 높음 (금액, 내역, 입금 예정일) | 즉시 구현 필수 |
| 11 | D-정산 | 정산 정보 입력 요청 | 광고주 정산 확정 (정보 미등록) | API hook (실시간) | 중 (금액, 마감일) | 즉시 구현 필수 |
| 12 | D-정산 | 입금 완료 확인 | 실제 입금 완료 처리 | API hook (실시간) | 높음 (금액, 계좌, 내역) | 즉시 구현 필수 |
| 13 | E-특수 | 가이드라인 위반 경고 | 위반 확정 처리 (사람 승인 필수) | 수동 트리거 | 높음 (위반 내역, 이의제기 기한) | 중요 |
| 14 | E-특수 | 탈퇴 처리 완료 | 탈퇴 신청 완료 | API hook (실시간) | 중 (미완료 정산, 설문 링크) | 중요 |
| 15 | F-거절 | 파트너 신청 거절 안내 | 광고주 거절 클릭 즉시 | API hook (실시간) | 높음 (프로그램명, 거절사유, 대안 프로그램) | 즉시 구현 필수 |

## 4. A/B 테스트 우선순위

| 순위 | 이메일 | 테스트 항목 | 이유 |
|------|--------|------------|------|
| **1** | 이메일 8: 첫 수익 확정 | 제목 A "감정 직격" vs B "숫자 강조" | 리텐션 결정적 순간. 개봉률 차이가 장기 활성율에 직결 |
| **2** | 이메일 4: 활동 미감지 넛지 | 제목 A "공감형" vs B "호기심 자극형" | 이탈 위험군 대상. 넛지 효과가 첫 활동 전환율을 좌우 |
| **3** | 이메일 10: 정산 확정 | 제목 A "금액 선제 공개" vs B "입금 예정일 강조" | 정산 이메일 개봉률이 CS 문의 감소 효과에 직결 |
| **4** | 이메일 5: 신규 프로그램 안내 | 제목 A "경쟁 심리" vs B "VIP 사전 안내" | 크로스셀 전환율. 파트너당 프로그램 수 증가에 기여 |
| **5** | 이메일 7: 첫 고객 유입 | 제목 A "직접적 축하" vs B "긴장감 자극" | 감정 자극 방식 차이가 이후 활동 지속률에 영향 |

### A/B 테스트 실행 기준
- 최소 표본: 각 그룹 50명 이상 (통계적 유의미성 확보)
- 측정 지표: 개봉률(primary), 클릭률(secondary), 후속 행동 전환율(tertiary)
- 테스트 기간: 최소 2주 (요일 편향 제거)
- 초기 파트너 수가 부족할 경우 A/B 테스트 보류, A안으로 단일 운영 후 데이터 축적

## 5. 구현 로드맵

### Phase 1: 즉시 구현 (1주 내)

**목표**: 파트너가 돈의 흐름을 확인할 수 있는 핵심 이메일 + 온보딩 업그레이드

| 이메일 | 난이도 | 구현 방식 | 비고 |
|--------|--------|-----------|------|
| **이메일 3**: 승인 완료 + 활동 가이드 | 낮음 | 기존 `sendPartnerApprovalEmail()` 업그레이드 | 기존 함수 있음. 템플릿 교체만 |
| **이메일 1**: 가입 축하 | 낮음 | 기존 `sendWelcomeEmail()` 업그레이드 | 기존 함수 있음. 템플릿 교체만 |
| **이메일 11**: 정산 정보 입력 요청 | 낮음 | 정산 확정 시 분기 추가 | 이탈 방지 직결. 돈 못 받으면 파트너 떠남 |
| **이메일 12**: 입금 완료 확인 | 낮음 | 입금 완료 처리 후 hook | CS 문의 즉시 감소 |
| **이메일 10**: 정산 확정 안내 | 낮음 | 이메일 11과 동일 트리거, 분기만 다름 | 이메일 11과 세트 구현 |
| **발송 로그 테이블** | 낮음 | `email_send_logs` DB migration | 전체 이메일의 중복 방지 기반 |

**Phase 1 예상 핵심 지표 개선**:
- 파트너 온보딩 완료율 +20~30% (이메일 1, 3 업그레이드 효과)
- 정산 관련 CS 문의 -40~50% (이메일 10, 11, 12 도입 효과)
- 정산 정보 미등록 파트너 비율 -30% (이메일 11 효과)

### Phase 2: 1~2주 내

**목표**: 넛지 시스템 + 성과 축하 이메일로 파트너 활성화

| 이메일 | 난이도 | 구현 방식 | 비고 |
|--------|--------|-----------|------|
| **이메일 7**: 첫 고객 유입 축하 | 낮음 | referral 생성 API 후처리 | 실시간 트리거 |
| **이메일 8**: 첫 수익 확정 축하 | 보통 | settlement 생성 후 "첫 건 여부" 체크 | 리텐션 핵심 |
| **이메일 2**: 프로그램 참여 독려 | 보통 | Cron 신규 개발 + 조건 쿼리 | 첫 번째 Cron 엔드포인트 |
| **이메일 4**: 활동 미감지 넛지 | 보통 | Cron 신규 개발 | 두 번째 Cron 엔드포인트 |

**Phase 2 예상 핵심 지표 개선**:
- 승인 후 7일 내 첫 활동 비율 +15~25% (이메일 4 넛지 효과)
- 가입 후 프로그램 신청 전환율 +10~20% (이메일 2 효과)
- 첫 리드 발생 후 30일 리텐션 +20% (이메일 7, 8 축하 효과)

### Phase 3: 1개월 내

**목표**: 자동화 완성 + 고급 기능 + 경계 상황 대응

| 이메일 | 난이도 | 구현 방식 | 비고 |
|--------|--------|-----------|------|
| **이메일 6**: 월간 성과 리포트 | 높음 | Cron + DB 집계 쿼리 + 분기 템플릿 | 가장 복잡. 프로그램별 KPI 분리 집계 |
| **이메일 13**: 가이드라인 위반 경고 | 보통 | Admin UI + violation_logs 테이블 신규 | 수동 확정 필수 로직 |
| **이메일 14**: 탈퇴 처리 완료 | 보통 | 탈퇴 API + 설문 JWT + 분기 렌더링 | 탈퇴 설문 수집 |
| **이메일 5**: 신규 프로그램 안내 | 보통 | 프로그램 활성화 트리거 + 수신자 필터링 | 대량 발송 큐 필요 |
| **이메일 9**: 티어 승급 축하 | 보통 | 티어 계산 로직 + 승급 감지 | 티어 시스템 선행 필요 |

**Phase 3 예상 핵심 지표 개선**:
- 월간 활성 파트너 비율(MAU) +10~15% (이메일 6 리포트 효과)
- 파트너당 평균 참여 프로그램 수 +0.3~0.5개 (이메일 5 크로스셀)
- 이탈 사유 데이터 축적 시작 (이메일 14 설문)
- 부정행위 재발률 감소 (이메일 13 경고 효과)

## 6. 검토 의견

### 6.1 전체 평가

5개 스트림의 이메일 템플릿은 파트너 라이프사이클을 빈틈없이 커버하고 있다. 특히 다음 부분이 우수하다:

- **정산 클러스터(D)의 분기 설계**: 정산 정보 등록 여부로 이메일 10/11을 깔끔하게 분기한 것은 실무적으로 매우 적절하다
- **성과 클러스터(C)의 스토리 흐름**: "첫 유입 → 첫 수익 → 티어 승급"이라는 성장 서사가 파트너 동기부여에 효과적
- **특수 상황(E)의 법적 고려**: 위반 경고 이메일에서 "사람 확정 필수" 원칙과 이의제기 채널 명시는 법적 리스크를 잘 방어

### 6.2 놓친 부분 및 개선 제안

#### (1) "프로그램 거절" 이메일 부재 -- 즉시 보완 필요

광고주가 파트너 신청을 "거절"하는 경우에 대한 이메일이 없다. 파트너 입장에서 신청 후 아무 소식이 없으면 플랫폼 신뢰도가 떨어진다. 거절 사유와 재신청 가이드를 포함한 **이메일 3-B (프로그램 신청 거절 안내)**를 추가해야 한다.

#### (2) 수신 거부(Unsubscribe) 메커니즘 미설계

한국 정보통신망법상 영리 목적 이메일에는 수신 거부 링크가 필수다. 14개 이메일 모두 푸터에 수신 거부 링크를 포함해야 하며, 수신 거부한 파트너를 발송 대상에서 제외하는 로직이 필요하다. 단, 정산 관련 이메일(10, 11, 12)과 가이드라인 위반(13)은 거래 관련 필수 통지이므로 수신 거부 대상에서 제외할 수 있다.

#### (3) 이메일 간 간섭 방지 로직 필요

예를 들어 파트너가 가입 후 3일째에 프로그램도 미신청이고 동시에 다른 이벤트가 겹치면 같은 날 이메일 2통 이상이 갈 수 있다. **일일 최대 발송 횟수 제한**(파트너당 1일 최대 2통) 또는 **쿨다운 기간**(이전 이메일 발송 후 24시간 내 Cron 기반 이메일 발송 억제) 규칙이 필요하다. 실시간 트리거 이메일(정산, 입금 등)은 쿨다운 예외로 처리한다.

#### (4) 기타 고려사항

- **이메일 열람 추적**: Resend API의 open/click tracking 활용하여 이메일 효과 측정. A/B 테스트 데이터 수집의 기반
- **다크 모드 호환**: 이메일 HTML 템플릿 작성 시 다크 모드 대응 필요 (배경색/텍스트색 미디어쿼리)
- **리드 상태 변경 알림 미포함**: 리드가 "유효 확정"되는 시점에 파트너에게 알리는 이메일이 없다. 이메일 7(첫 유입)과 이메일 8(첫 수익) 사이의 간극이 크므로, "리드 유효 확정" 알림 추가를 검토할 것
- **광고주 대상 이메일**: 현재 14개는 모두 파트너 대상. 광고주에게 "새 파트너 신청 알림", "리드 유입 알림" 등도 별도 스트림으로 기획 필요

## 7. 이메일 발송 제한 규칙 (Throttle)

> 상세 기술 스펙: `docs/crm/G-throttle-spec.md`

### 쿨다운 규칙 요약

- **마케팅/참여 이메일**: 파트너당 1일 최대 **2통** (KST 자정 기준 리셋)
- **동일 이메일 중복 방지**: 동일 이메일 유형은 24시간 내 재발송 불가
- **Cron 이메일 이월**: 일일 한도 초과 시 우선순위 낮은 Cron 이메일은 다음 날 자동 이월. 별도 큐 없이 다음 Cron 실행 시 조건 재판단.
- **우선순위 충돌**: 성과(1순위) → 온보딩(2순위) → 활동(3순위) → Cron(4순위). 한도 초과 시 낮은 순위가 이월.

### 예외 (즉시 발송, 쿨다운 미적용)

| 이메일 | 발송 유형 |
|--------|----------|
| 이메일 10: 정산 확정 안내 | 즉시 발송 필수 (금전 정보) |
| 이메일 11: 정산 정보 입력 요청 | 즉시 발송 필수 (금전 정보) |
| 이메일 12: 입금 완료 확인 | 즉시 발송 필수 (금전 정보) |
| 이메일 13: 가이드라인 위반 경고 | 즉시 발송 필수 (법적 고지) |
| 이메일 14: 탈퇴 처리 완료 | 즉시 발송 필수 (계정 상태) |
| 이메일 15: 파트너 신청 거절 | 즉시 발송 필수 (신청 결과) |

> 예외 이메일은 일일 2통 한도 카운트에도 포함되지 않는다. 정산 이메일이 나간 날에도 Cron 이메일 2통 한도는 별도로 유지된다.

### 필수 인프라: email_log 테이블

| 필드 | 역할 |
|------|------|
| `partner_id` | 수신자 파트너 UUID |
| `email_type` | 이메일 유형 식별자 (예: `onboarding_nudge`) |
| `sent_at` | 발송 일시 (UTC). 24h 중복 판단 기준 |
| `is_mandatory` | true = 즉시 발송 예외. 쿨다운 카운트 미포함 |
| `program_id` | nullable. 프로그램 단위 이메일에만 사용 |
| `status` | `sent` / `deferred` / `failed` |

`email_log` 테이블은 **전체 15종 이메일의 중복 방지 및 쿨다운 판단의 공통 기반**이다. Phase 1 구현 시작 전에 migration을 먼저 적용해야 한다.
