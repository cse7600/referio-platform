# Referio 플랫폼 온보딩 가이드

> **버전**: 1.1 | **작성일**: 2026-03-27 | **대상**: 기획자/운영자/CRM 마케터
> 이 문서 하나로 Referio 플랫폼의 전체 구조, 기능, 운영 방법을 파악할 수 있습니다.

---

## 0. TL;DR — 30초 요약

Referio는 **B2B 기업(광고주)이 추천인(파트너)을 통해 잠재 고객을 유치하고, 성과에 따라 수수료를 자동 정산하는 어필리에이트 플랫폼**입니다. 파트너가 추천 링크를 공유하면 잠재 고객이 문의를 남기고, 광고주가 영업하여 계약이 성사되면 파트너에게 수수료가 자동 생성됩니다. 현재 한화비전 키퍼메이트(파트너 99명)와 밀리의서재(앱 이벤트 트래킹)가 연동되어 있으며, Airtable 기반 CRM 연동을 지원합니다.

## 1. 서비스 개요

### 1.1 Referio란 무엇인가?

Referio는 **"소개비 자동 정산 플랫폼"**입니다.

쉽게 말하면:
- A회사(광고주)가 "고객을 데려오면 수수료를 주겠다"고 합니다
- 블로거 B(파트너)가 자신의 블로그에 A회사 추천 링크를 올립니다
- 독자 C가 그 링크를 통해 A회사에 문의합니다
- A회사가 C와 계약을 맺으면 블로거 B에게 수수료가 자동 정산됩니다

Referio는 이 과정 전체(링크 발급 → 리드 추적 → 계약 관리 → 수수료 정산)를 하나의 플랫폼에서 자동화합니다.

**서비스 URL**:
| 항목 | URL |
|------|-----|
| 서비스 홈 | https://referio.kr |
| 광고주 로그인 | https://referio.kr/advertiser/login |
| 파트너 로그인 | https://referio.kr/login |
| 관리자 패널 | https://referio.kr/admin |

### 1.2 핵심 가치 제안 (Why Referio?)

**광고주 입장**:
- 영업 직원 없이도 파트너 채널을 운영할 수 있음
- 성과 기반 정산이므로 비용 효율이 극대화됨
- 리드 유입부터 영업 파이프라인까지 CRM 없이 관리 가능

**파트너 입장**:
- 추천 링크 하나로 수익 창출
- 실시간으로 성과를 추적할 수 있음
- 티어(등급) 시스템으로 장기 성장 동기 부여

### 1.3 비즈니스 모델

Referio의 수익원은 두 가지입니다:

1. **브랜디드 콘텐츠 협업 수수료**: 광고주가 파트너에게 콘텐츠 제작을 의뢰할 때 예산의 10%를 플랫폼 수수료로 수취
2. **커스텀 계약 기반 월정액**: 광고주별 맞춤 요금으로 운영 중 (정가 공개 보류, 별도 문의로 운영)

> 참고: 요금제 페이지(`/advertiser/plan`)는 현재 "별도 문의 안내" 페이지로 교체되어 있습니다.

### 1.4 타겟 시장 & 고객

**1차 타겟 (광고주)**:
- 국내 B2B SaaS, IT서비스, 기업 솔루션, 금융 서비스 등을 운영하는 중소기업
- 영업인력이 부족하고 파트너 채널을 구축하고 싶지만 관리 도구가 없는 회사

**2차 타겟 (파트너)**:
- 비즈니스/기술 분야 블로거, 유튜버, 인스타그래머
- 기업 컨설턴트, 에이전시
- 지인 네트워크 보유자 (B2B 서비스 추천으로 수익을 원하는 사람)

## 2. 주요 이해관계자 (Stakeholders)

### 2.1 광고주 (Advertiser)

**누구인가**: 파트너 채널을 통해 고객을 유치하고 싶은 B2B 기업 (예: ERP 소프트웨어 회사, HR SaaS, IT 솔루션 업체)

**무엇을 원하는가**:
- 파트너를 모집하고 관리
- 파트너가 데려온 잠재 고객(리드)을 영업 파이프라인으로 관리
- 계약 완료 시 파트너에게 수수료를 정산

**어떻게 쓰는가**:
1. Referio에 광고주 계정을 만든다
2. 캠페인(수수료 단가)을 설정한다
3. 파트너 신청을 승인/거절한다
4. 들어온 리드를 영업 관리한다 (통화 기록, 상태 변경)
5. 계약 완료 처리하면 정산이 자동 생성된다
6. 실제 송금 후 정산 완료 버튼을 누른다

**로그인 방식**: 광고주 ID + 사용자 ID + 비밀번호 (3단계)
- 하나의 광고주 계정에 여러 사용자가 존재 가능 (admin/manager/viewer 역할)

### 2.2 파트너 (Partner)

**누구인가**: 추천 링크를 공유해서 수수료를 받는 개인 또는 팀 (예: 블로거, 유튜버, 컨설턴트, 에이전시)

**무엇을 원하는가**:
- 추천 활동으로 수익 창출
- 실시간 성과 확인 (내가 유치한 고객, 수수료)
- 투명한 정산

**어떻게 쓰는가**:
1. Referio에 파트너 계정을 만든다 (이메일/비밀번호)
2. 프로그램 마켓플레이스에서 원하는 광고주의 프로그램에 신청한다
3. 광고주가 승인하면 나만의 추천 링크가 발급된다
4. 추천 링크를 블로그/SNS/카카오톡 등에 공유한다
5. 독자가 링크를 클릭하고 문의를 남기면 리드가 자동 기록된다
6. 대시보드에서 유치 고객, 유효 건수, 정산 금액을 확인한다

**로그인 방식**: 이메일 + 비밀번호 (Supabase Auth 기반)

### 2.3 Referio 운영팀 (Admin)

**역할**:
- 전체 광고주/파트너 현황 모니터링
- 이슈 발생 시 데이터 직접 수정 (Supabase Dashboard 활용)
- 신규 광고주 온보딩 지원

**접근 방법**: `/admin` 페이지 (app_metadata.role=admin 기반 인증)
- 마스터/어드민 역할 구분
- 어드민 액세스 초대/취소 기능 (`/admin/users`)

## 3. 핵심 개념 정의 (용어 사전)

| 용어 | 영어/DB 컬럼 | 설명 |
|------|-------------|------|
| **광고주** | advertiser | 파트너를 통해 고객을 유치하려는 B2B 기업 |
| **파트너** | partner | 추천 링크를 공유해 수수료를 받는 개인/팀 |
| **광고주 ID** | advertiser_id (text) | 광고주 계정을 식별하는 텍스트 (예: `keepermate`, `millie`) |
| **광고주 UUID** | advertisers.id (UUID) | 시스템 내부에서 사용하는 고유 식별자. 기획/운영 시 직접 다룰 일은 적음 |
| **파트너 프로그램** | partner_programs | 광고주와 파트너를 연결하는 중개 테이블. 하나의 파트너가 여러 광고주 프로그램에 동시 참가 가능 |
| **추천 코드** | referral_code | 파트너별로 발급되는 고유 식별 코드 (예: `ABC123`). 프로그램마다 다르게 발급됨 |
| **추천 링크** | referral_url | 파트너가 공유하는 URL. `?ref=추천코드` 형태로 파트너를 식별 |
| **리퍼럴 (리드)** | referral | 파트너 링크를 통해 들어온 잠재 고객 문의 1건 |
| **계약 상태** | contract_status | 리드의 현재 영업 단계: pending(신규) → call_1(1차통화) → call_2 → call_3 → completed(계약완료) 또는 invalid(무효)/duplicate(중복) |
| **유효 처리** | is_valid | 광고주가 "이 리드는 진짜 잠재 고객"이라고 확인한 표시 |
| **정산** | settlement | 파트너에게 지급하는 수수료 내역. 유효 정산(valid)과 계약 정산(contract) 두 종류 |
| **정산 상태** | settlement.status | pending(대기) → completed(완료) 또는 rejected(거절) |
| **캠페인** | campaign | 수수료 단가 및 정책을 정의하는 광고주 설정. 현재 광고주당 1개 |
| **유효 단가** | valid_amount | 리드 1건이 유효 처리될 때 파트너에게 지급하는 금액 (예: 5만원) |
| **계약 단가** | contract_amount | 계약 1건이 완료될 때 파트너에게 지급하는 금액 (예: 10만원) |
| **파트너별 개별 단가** | lead_commission / contract_commission | 특정 파트너에게만 다른 단가를 적용할 때 사용하는 설정 (파트너 프로그램 단위) |
| **티어** | tier | 파트너 등급: Authorized(기본) → Silver → Gold → Platinum. 등급별로 단가 차등 가능 |
| **브랜디드 콘텐츠 협업** | collaboration | 광고주가 우수 파트너에게 콘텐츠 제작을 의뢰하는 기능. 플랫폼 수수료 10% |
| **이벤트/프로모션** | promotion | 광고주가 파트너 대상으로 만드는 인센티브 행사 (예: "이번 달 TOP 파트너 보너스") |
| **웹훅** | webhook | 외부 서비스(Airtable, CRM 등)와 데이터를 자동으로 주고받는 기술 연결 통로 |
| **트래킹 링크** | tracking_link | 밀리의서재 같은 B2C 광고주용. 에어브릿지 등 외부 서비스에서 발급하는 앱 설치 추적 전용 링크 |
| **이벤트 트래킹** | event_tracking | B2C 방식의 전환 추적. 앱 설치 → 가입 → 구독 같은 퍼널 단계별 이벤트를 자동 수신 |
| **광고주 유형** | advertiser_type | inquiry(B2B 문의 기반, 기본), event_tracking(B2C 이벤트 기반), hybrid(둘 다) |
| **Referio 자체 추천 프로그램** | affiliate_campaign | Referio가 직접 운영하는 파트너/광고주 모집 캠페인. 파트너가 다른 파트너나 광고주를 데려오면 보상 |
| **마켓플레이스** | /dashboard/programs | 파트너가 참가 가능한 프로그램 목록을 보는 페이지. 광고주 모집 캠페인은 숨겨지고 파트너 모집 프로그램만 표시 |

## 4. 서비스 전체 플로우

### 4.1 광고주 온보딩 플로우

```
1. https://referio.kr/advertiser/signup 에서 회원가입
   - 광고주 ID 설정 (예: puzl_co_kr)
   - 사용자 ID, 비밀번호, 회사명 입력
         ↓
2. 대시보드에 3단계 시작 가이드 표시 (파트너 0명일 때)
   - ① 캠페인 설정 → ② 파트너 모집 → ③ 리드 관리
         ↓
3. 캠페인 설정 (/advertiser/campaigns)
   - 유효 단가, 계약 단가 입력
   - 중복 체크 기간, 유효 판정 기한, 계약 완료 기한 설정
         ↓
4. 파트너 모집
   - 파트너 초대 링크 복사 → SNS/이메일로 공유
   - 또는 브랜딩된 가입 페이지: /signup/{광고주ID}
         ↓
5. 파트너 신청 처리 (/advertiser/partners)
   - 대기 중 탭에서 신청 확인 → 승인 또는 거절
   - 승인 시 파트너에게 추천 코드 자동 발급 + 이메일 알림
         ↓
6. 리드 관리 (/advertiser/referrals)
   - 파트너 링크를 통해 들어온 리드가 자동 표시
   - 영업 상태 변경: 신규 → 1차통화 → 2차통화 → 3차통화 → 계약완료
   - 유효 처리(is_valid=true) 시 유효 정산 자동 생성
   - 계약 완료(completed) 시 계약 정산 자동 생성
         ↓
7. 정산 관리 (/advertiser/settlements)
   - 대기 중 탭에서 정산 건 확인
   - 실제 계좌이체로 송금 후 "완료 처리" 버튼 클릭
   - 개별 처리 또는 체크박스 선택 후 일괄 처리 가능
```

### 4.2 파트너 온보딩 플로우

```
1. https://referio.kr/signup 에서 파트너 가입
   - 이메일, 비밀번호 입력 → Supabase Auth 계정 생성
   - 가입 완료 시 환영 이메일 자동 발송
         ↓
2. 온보딩 (/onboarding)
   - 프로필 설정
         ↓
3. 프로그램 탐색 (/dashboard/programs)
   - 마켓플레이스에서 참가 가능한 프로그램 확인
   - 원하는 프로그램 "참가 신청" 클릭
   - 자동 승인 설정된 광고주는 즉시 승인, 아니면 대기
         ↓
4. 승인 완료 시
   - 승인 알림 이메일 수신 (추천 코드, 링크, 커미션 안내 포함)
   - 대시보드에 추천 URL 표시
         ↓
5. 추천 활동
   - 추천 URL을 블로그/SNS/카카오톡 등에 공유
   - 독자가 링크 클릭 → 문의 폼 작성 → 리드 자동 기록
         ↓
6. 성과 확인 (/dashboard)
   - 유치 고객 수, 유효 건수, 계약 건수, 누적 정산 금액 확인
   - 이번 달 내 순위, 마일스톤 진행률 확인
```

### 4.3 리퍼럴 발생 플로우 (추천 링크 → 정산까지)

```
파트너가 추천 링크 공유
  https://referio.kr/inquiry/{광고주ID}?ref={추천코드}
         ↓
잠재 고객이 링크 클릭 → 문의 폼 페이지 도착
  - 광고주 브랜딩 적용 (로고, 색상, 프로그램 설명)
         ↓
이름, 연락처, 문의내용 입력 → 제출
         ↓
POST /api/inquiry 호출
  - ?ref= 파라미터의 추천코드로 partner_programs 테이블에서 파트너 자동 식별
  - 같은 광고주 + 같은 연락처로 90일 이내 중복 문의 차단
  - referrals 테이블에 리드 생성 (contract_status: 'pending')
         ↓
광고주에게 이메일 + Slack 알림
  "[Referio] 새 리드 접수 — 홍길동"
  파트너에게도 이메일 알림
         ↓
광고주 리드 관리 화면에 즉시 표시
         ↓
광고주가 영업 파이프라인으로 관리
  pending → call_1 → call_2 → call_3 → completed
         ↓
[유효 처리 시] is_valid = true 설정
  → 정산 자동 생성 (type: valid, amount: 유효 단가)
         ↓
[계약 완료 시] contract_status = completed 설정
  → 정산 자동 생성 (type: contract, amount: 계약 단가)
         ↓
광고주가 실제 송금 후 정산 상태를 'completed'로 변경
  → 파트너 대시보드에 완료 표시
```

### 4.4 Airtable 연동 플로우

광고주가 이미 Airtable로 리드를 관리하고 있는 경우의 연동 방식입니다.

**방식 1: Airtable → Referio 자동 푸시 (Automation)**

```
광고주 웹사이트 문의폼에 추천코드 숨김 필드 추가
  → ?ref=코드를 URL에서 자동 캡처
         ↓
문의 제출 → Airtable에 레코드 생성 (추천코드 포함)
         ↓
Airtable Automation이 트리거 발동
  → POST /api/webhook/airtable (X-API-Key 헤더 포함)
         ↓
Referio가 리드 생성 + 파트너 자동 매칭
         ↓
영업 진행 → Airtable에서 영업상태 변경 (유효/계약완료)
         ↓
Airtable Automation 재발동 → Referio에 상태 변경 전송
         ↓
Referio가 리퍼럴 상태 업데이트 → 정산 자동 생성
```

**방식 2: Referio → Airtable 주기적 폴링 (Cron)**

```
Referio 크론잡이 30분마다 Airtable API 조회
  → /api/cron/sync-airtable-partners
         ↓
신규 파트너 데이터 감지
  → Supabase Auth 계정 자동 생성
  → partner_programs 등록
  → 초대 이메일 자동 발송 (설정 시)
```

**광고주 설정 방법 (4단계 위저드)**:
| 단계 | 내용 |
|------|------|
| 1. 연동 확인 | 웹훅 URL과 API 키 확인 |
| 2. 필드 설정 | Airtable 컬럼명과 Referio 필드 매핑 (이름, 전화번호, 추천코드, 영업상태) |
| 3. 웹사이트 설치 | 추천코드 캡처 JS 스크립트 복사 → 광고주 사이트에 삽입 |
| 4. Automation 설정 | Airtable에 붙여넣을 스크립트 코드 복사 |

#### 5.2.1 CRM 운영 필드 상세

리드(리퍼럴) 하나하나에 다음과 같은 운영 관리 필드가 있습니다. 광고주가 리드를 영업 관리할 때 활용합니다.

| 필드명 | 기능 | 실제 활용 예시 |
|--------|------|---------------|
| **영업 담당자** (sales_rep) | 이 리드를 누가 담당하는지 배정 | "김팀장", "이대리" 등 이름 입력. 팀이 여러 명일 때 리드를 나눠서 관리 |
| **우선순위** (priority) | 리드의 긴급도를 4단계로 표시 | `low`(낮음), `normal`(보통), `high`(높음), `urgent`(긴급) — 대기업 문의는 urgent로 설정 |
| **라벨** (labels) | 자유 태그를 여러 개 붙일 수 있음 | `#대기업`, `#급함`, `#재문의`, `#VIP` 등 — 검색이나 분류에 활용 |
| **다음 할 일** (next_action) | 이 리드에 대해 다음에 해야 할 일을 메모 | "견적서 발송", "미팅 일정 잡기", "상품 안내 자료 보내기" 등 |
| **다음 액션 예정일** (next_action_at) | 다음 할 일의 마감일/예정일 | 3일 후 날짜를 넣어두면 잊지 않고 follow-up 가능 |
| **내부 메모** (memo) | 내부 영업 노트 (파트너에게 보이지 않음) | "통화 결과: 예산 500만원, 4월 중 결정 예정", "담당자 부재 — 화요일 재연락" |

**활용 팁 (CRM 마케터/운영자용)**:

1. **영업 담당자 배정 워크플로우**: 리드가 들어오면 즉시 `sales_rep`에 담당자를 입력합니다. 팀원이 여러 명이면 돌아가며 배정하거나, 지역/분야별로 나눠 배정합니다.

2. **우선순위 관리 예시**:
   - `urgent`: 대기업 문의, 당일 연락 필수
   - `high`: 예산 규모가 큰 리드, 경쟁사 전환 문의
   - `normal`: 일반 문의 (대부분의 리드)
   - `low`: 단순 정보 요청, 장기 검토 중

3. **라벨 태그 활용 예시**:
   - `#대기업` — 대기업 리드만 모아보기
   - `#급함` — 빠른 대응이 필요한 건
   - `#재문의` — 이전에 연락이 왔었던 고객
   - `#견적발송` — 견적서를 보낸 상태
   - `#컨퍼런스` — 행사에서 만난 리드

4. **다음 할 일 + 예정일 활용**: 매일 아침 "오늘 예정일인 리드"를 확인하면 팔로업 누락을 방지할 수 있습니다. 예: "3월 28일까지 견적서 발송" → 해당 날짜에 리드를 확인하고 처리.

#### 5.2.2 리드 상태(contract_status) 전환 규칙

리드의 계약 상태는 7가지이며, 각각 의미와 전환 시점이 명확합니다.

| 상태 | 표시 이름 | 의미 | 언제 전환하나? |
|------|----------|------|-------------|
| `pending` | 대기 | 방금 들어온 리드. 아직 연락을 시도하지 않은 상태 | 리드가 처음 생성될 때 자동 설정 |
| `call_1` | 1차 콜 | 첫 번째 통화를 완료한 상태 | 고객에게 첫 전화를 걸고 대화를 나눈 후 |
| `call_2` | 2차 콜 | 두 번째 통화를 완료한 상태 | 추가 논의, 견적 안내 등 후속 통화 후 |
| `call_3` | 3차 콜 | 세 번째 이상 통화를 한 상태 | 심화 논의, 계약 조건 조율 등 |
| `completed` | 계약완료 | 계약이 체결된 상태 | 실제 계약서 체결 또는 구매 확정 후 |
| `invalid` | 무효 | 영업 가치가 없는 리드 | 연락 불가, 관심 없음, 자격 미달 등 |
| `duplicate` | 중복 | 이미 존재하는 고객 | 기존 고객, 이미 다른 파트너가 소개한 고객 등 |

**상태 전환 시 주의사항**:

- **무효/중복 처리와 유효 처리는 별개입니다**: 계약 상태를 `invalid`(무효)로 바꿔도 `is_valid` 값은 변하지 않습니다. `is_valid`는 "이 리드가 진짜 잠재 고객인지"를 표시하는 것이고, 계약 상태는 "영업이 어디까지 진행됐는지"를 표시하는 것입니다.
- **정산은 오직 `is_valid = true`일 때만 생성됩니다**: 무효나 중복으로 표시하면 `is_valid`는 null(미확인) 상태로 남아서 정산이 발생하지 않습니다.
- **계약완료(completed)로 바꾸면 자동으로 유효 처리됩니다**: 계약이 됐다는 것은 당연히 유효한 리드이므로, `is_valid`가 자동으로 `true`로 바뀝니다 (상세 로직은 13장 참고).
- **상태를 되돌릴 수 있습니다**: 실수로 상태를 바꾼 경우, 다시 이전 상태로 변경 가능합니다. 단, 이미 생성된 정산은 자동 삭제되지 않으므로 수동 확인이 필요합니다.

- 파트너 초대 링크 복사 기능

**파트너 티어(등급)**:
| 티어 | 의미 | 수수료 |
|------|------|--------|

### 5.4 정산 관리 — `/advertiser/settlements`

**통계 카드**: 대기 건수/금액, 완료 건수/금액, 이번 달 정산, 선택된 금액

**대기 탭**:
- 아직 송금하지 않은 정산 건 목록
- 개별 완료 처리 또는 체크박스로 선택하여 일괄 완료 처리
- 정산 유형 표시: 유효 정산(valid) / 계약 정산(contract)

**완료 탭**:
- 정산 이력 (정산일, 파트너명, 고객명, 유형, 금액, 비고)
- 고객명은 첫 글자만 표시 (개인정보 보호)

**CSV 내보내기**: 정산 데이터를 엑셀로 다운로드 가능

### 5.5 캠페인 설정 — `/advertiser/campaigns`

수수료 정책을 정의하는 곳입니다.

**기본 설정**:
| 항목 | 설명 | 예시 |
|------|------|------|
| 캠페인 활성화/비활성화 | 캠페인을 켜고 끌 수 있음 | — |
| 유효 단가 (valid_amount) | 리드 1건이 유효 처리될 때 파트너에게 지급 | 5만원 |
| 계약 단가 (contract_amount) | 계약 1건 완료 시 지급 | 10만원 |
| 수수료율 (commission_rate) | 정률 방식 시 사용 | 5% |
| 최소 정산 금액 | 이 금액 이하는 정산 대기 | 1만원 |
| 중복 체크 기간 | 같은 연락처 재유입 방지 기간 | 90일 |
| 유효 판정 기한 | 문의 후 유효 처리 목표일 | 7일 |
| 계약 완료 기한 | 계약 완료 목표일 | 30일 |

**티어별 단가**: Authorized/Silver/Gold/Platinum 각각 다른 유효단가/계약단가 설정 가능

**파트너별 개별 단가 설정**:

캠페인의 기본 단가 외에, 특정 파트너에게만 다른 단가를 적용할 수 있습니다.

`partner_programs` 테이블에서 다음 두 필드를 사용합니다:
- `lead_commission` — 해당 파트너의 개별 유효 단가 (설정 시 캠페인 기본 단가 대신 적용)
- `contract_commission` — 해당 파트너의 개별 계약 단가 (설정 시 캠페인 기본 단가 대신 적용)

**정산 단가 적용 우선순위** (중요):
1. 1순위: 파트너별 개별 단가 (`partner_programs.lead_commission` / `contract_commission`)가 설정되어 있으면 이 값을 사용
2. 2순위: 개별 단가가 없으면 캠페인 기본 단가 (`campaigns.valid_amount` / `contract_amount`)를 사용

**활용 예시**: 성과가 뛰어난 파트너 A에게는 유효 단가 7만원(기본 5만원보다 높음)을 설정하여 추가 인센티브를 제공할 수 있습니다. 나머지 파트너들은 캠페인 기본 단가 5만원이 적용됩니다.

> 설정을 변경하면 **이후 생성되는 리드**부터 적용됩니다. 기존 리드에는 영향 없습니다.
> 현재 광고주당 캠페인 1개만 지원합니다. 신규 캠페인 생성 버튼은 있지만 "준비 중" 상태입니다.

### 5.6 이벤트/프로모션 관리 — `/advertiser/promotions`

파트너 대상 인센티브 행사를 만드는 기능입니다.

**이벤트 생성**: 제목, 유형(이벤트/보너스/랭킹), 리워드 설명, 기간, 상세 내용, 파트너 공개 여부
**이벤트 목록**: 진행중/종료/임시저장 상태 관리
**파트너 노출**: `is_visible_to_partners = true`인 이벤트는 파트너 대시보드에 배너로 자동 노출

### 5.7 브랜디드 콘텐츠 협업 — `/advertiser/collaborations`

우수 파트너에게 콘텐츠 제작을 의뢰하는 기능입니다.

**파트너 탐색**: 계약 실적 상위 파트너 목록 + 협업 제안 버튼
**협업 제안**: 파트너 선택, 콘텐츠 유형(블로그/유튜브/인스타/틱톡/기타), 예산, 마감일, 브리프
**플랫폼 수수료**: 예산의 10%를 자동 계산하여 표시 (파트너 수취분 = 예산 - 10%)
**협업 상태 흐름**:
```
제안(requested) → 수락(accepted) → 진행중(in_progress) → 제출(submitted)
  → 수정요청(revision) 또는 완료(completed) → 지급(paid)
  또는 거절(declined) / 취소(cancelled)
```
**메시지 채팅**: 협업 상세에서 광고주-파트너 간 메시지 주고받기 가능

### 5.8 활동 지원 게시판 — `/advertiser/activity-support`

파트너에게 공지사항이나 콘텐츠 소스를 제공하는 기능입니다.

**공지사항 탭**: 중요 안내를 작성하면 파트너 대시보드에 노출
**게시판 탭**: 일반 게시물 작성 (콘텐츠 소재, 참고 자료 등)
**에디터**: 노션 스타일 Tiptap 에디터 — 서식, 이미지 업로드/붙여넣기 지원
**게시물 수정**: 인라인 수정 모드로 제목, 내용 즉시 수정 가능
**공개/비공개**: 게시물별 파트너 공개 여부 설정

### 5.9 리포트 — `/advertiser/reports`

파트너 실적 보고서를 작성, 저장, PDF로 다운로드하는 기능입니다.

**에디터**: 노션 스타일 Tiptap 에디터
**실시간 통계 위젯**: 에디터에 삽입 가능한 데이터 블록 — 파트너 현황, 리퍼럴 현황, 정산 현황, TOP 파트너 등
**PDF 다운로드**: 작성한 보고서를 PDF로 내보내기
**리포트 저장/목록**: 작성한 보고서를 저장하고 목록에서 조회
**템플릿**: 한화비전 키퍼메이트 보고서 템플릿 등 미리 제공

### 5.10 메시지 — `/advertiser/messages`

파트너 전체에게 공지 메시지를 발송하는 기능입니다.
- 대상: 전체 파트너, 특정 티어, 개별 파트너 지정 가능
- 읽음 수 확인

### 5.11 이벤트 현황 — `/advertiser/events`

B2C 이벤트 트래킹 방식 광고주(예: 밀리의서재) 전용 페이지입니다.

**퍼널 요약 카드**: 설치/가입/구독 건수 + 전환율
**파트너별 이벤트 테이블**: 설치/가입/구독/전환율/정산금액
**최근 이벤트 스트림**: 최근 20건

> 사이드바에서 advertiser_type이 `event_tracking` 또는 `hybrid`인 광고주에게만 "이벤트 현황" 메뉴가 표시됩니다.

### 5.12 설정 — `/advertiser/settings`

**비밀번호 변경**: 현재 비밀번호 → 새 비밀번호

**프로그램 미디어 관리**: 이미지/동영상/유튜브 첨부 관리
**파트너 가입 링크**: 브랜딩된 가입 URL 확인

### 5.13 요금제 — `/advertiser/plan`

## 7. 현재 연동된 외부 서비스 & 인프라

### 7.1 Supabase (데이터베이스 + 인증)

Referio의 모든 데이터를 저장하는 곳입니다. PostgreSQL 기반의 클라우드 데이터베이스 서비스입니다.

**용도**:
- 모든 테이블 데이터 저장 (광고주, 파트너, 리퍼럴, 정산 등)
- 파트너 인증 (Supabase Auth — 이메일/비밀번호)
- RLS(Row Level Security)로 데이터 접근 제어

**RLS 정책 요약**:
- 파트너는 자기 자신의 데이터만 조회 가능
- 광고주 데이터는 RLS로 차단 (광고주는 별도 쿠키 기반 인증 사용)
- 웹훅 설정은 모든 일반 접근 차단 (관리자만 접근)
- 관리자 API는 service_role 키로 RLS를 우회

**프로젝트 정보**:
- 프로젝트 ref: `eqdnirtgmevhobmycxzn`
- Dashboard: https://supabase.com (프로젝트 선택)

### 7.2 Vercel (배포)

Referio 서비스를 인터넷에 공개하는 호스팅 플랫폼입니다.

**동작 방식**: GitHub의 `main` 브랜치에 코드를 푸시하면 자동으로 새 버전이 배포됩니다.
**GitHub 저장소**: `cse7600/referio-platform`

### 7.3 Airtable 연동

광고주가 Airtable로 고객 데이터를 관리할 때 Referio와 자동 동기화하는 기능입니다.

**웹훅 엔드포인트**: `POST /api/webhook/airtable`
**인증**: X-API-Key 헤더 (필수), X-Timestamp (권장, 5분 윈도우), X-Signature (선택, HMAC-SHA256)
**멱등성**: `airtable_record_id` 기반 중복 방지 (같은 레코드를 여러 번 보내도 한 번만 처리)
**파트너 자동 싱크 Cron**: `/api/cron/sync-airtable-partners` — 30분 주기로 Airtable에서 파트너 데이터를 읽어와 자동 등록

### 7.4 Airbridge 연동 (밀리의서재)

밀리의서재 같은 B2C 앱 서비스 전용 이벤트 트래킹 연동입니다.

**웹훅 엔드포인트**: `POST /api/webhook/airbridge`
**퍼널 단계**: 설치(install) → 가입(sign_up) → 구독(subscribe)
- install: 트래킹 이벤트만 기록 (리퍼럴 생성 안 함)
- sign_up: 리퍼럴 자동 생성 (고객 이름/연락처 없이 device_id로 식별)
- subscribe: 기존 리퍼럴을 계약완료로 변경 → 정산 자동 생성

**파트너 식별**: `sub_id` 파라미터로 트래킹 링크 → 파트너 매핑

### 7.5 Resend (이메일 발송)

Referio에서 자동 발송하는 이메일 서비스입니다.

**발신 주소**: `noreply@updates.puzl.co.kr`
**발신자 이름**: Referio
**무료 플랜 한도**: 월 3,000건 / 일 100건

**현재 운영 중인 이메일 (3종)**:
| 이메일 | 트리거 | 수신자 |
|--------|--------|--------|
| 파트너 가입 환영 | 파트너 최초 가입 시 | 신규 파트너 |
| 파트너 승인 알림 | 광고주가 파트너 승인 시 | 승인된 파트너 |
| 광고주 새 리드 알림 | 문의폼/Airtable로 리드 생성 시 | 광고주 담당자 |

**구현 예정 이메일 (5종)**: 리드 유효 확정 알림, 파트너 신청 알림, 계약 완료 알림, 정산 처리 알림, 비활성 리마인더

### 7.6 GTM (Google Tag Manager)

웹사이트 방문자 행동 추적을 위한 구글 태그 관리자입니다.
- 코드: `GTM-K7NZSXZ3`
- 적용 위치: `layout.tsx` (전체 페이지)
- 문의폼 제출 시 GTM 이벤트 발생 (`trackInquirySubmit`)

### 7.7 Slack 알림

주요 이벤트 발생 시 Slack 채널에 자동 알림을 보냅니다.
- 새 문의 접수 시 (`notifyNewInquiry`)
- Airtable 리드 접수 시 (`notifyAirtableLead`)
- 파트너 프로그램 신청 시 (`notifyPartnerApply`)

### 7.8 Referio 어필리에이트 트래킹

Referio 자체 광고/마케팅 성과를 추적하는 시스템입니다. 파트너 추천 링크(inquiry)와는 별개로, Referio 회사가 직접 운영하는 마케팅 캠페인의 클릭과 전환을 추적합니다.

**구성 요소**:
- `referio_affiliate_links` — 단축 URL을 관리하는 테이블. 각 링크에 고유 코드와 목적지 URL이 있음
- `referio_affiliate_events` — 클릭/전환 이벤트를 기록하는 테이블
- `referio_campaigns` — Referio 자체 마케팅 캠페인 관리

**동작 방식**:
1. **클릭 추적**: `/api/r/[code]` — 단축 URL을 클릭하면 클릭 이벤트가 기록되고, 설정된 랜딩 페이지로 자동 이동합니다
2. **전환 추적**: `/api/affiliate/convert` — 랜딩 페이지에서 전환(회원가입, 문의 등)이 발생하면 이 API를 호출하여 전환 이벤트를 기록합니다

**관리 화면**: `/admin/affiliate` — 관리자 패널에서 링크 생성, 클릭 수, 전환 수 등을 확인할 수 있습니다

**활용 예시**: Referio가 광고주 모집을 위해 블로그 글에 단축 URL을 삽입 → 클릭 수 추적 → 회원가입 전환 추적

## 13. 알려진 이슈 & 운영 주의사항

### 정산 자동 생성 로직 상세 (운영자 필수 이해)

정산(수수료)은 수동으로 생성하는 것이 아니라, 리드의 상태를 바꾸면 **자동으로 생성**됩니다. 운영 실수를 방지하기 위해 아래 로직을 반드시 이해해야 합니다.

**시나리오 1: "유효" 버튼을 누르면 (`is_valid = true`)**
- 유효 정산(type: valid)이 자동으로 1건 생성됩니다
- 이때 정산 금액은 아래 우선순위로 결정됩니다:
  - 1순위: 해당 파트너에게 개별 단가(`partner_programs.lead_commission`)가 설정되어 있으면 그 금액
  - 2순위: 개별 단가가 없으면 캠페인 기본 유효 단가(`campaigns.valid_amount`)

**시나리오 2: "계약완료"로 상태를 바꾸면 (`contract_status = completed`)**
- 먼저 `is_valid`가 자동으로 `true`로 바뀝니다 (계약이 됐으니 당연히 유효한 리드이므로)
- 유효 정산(type: valid)이 아직 없으면 자동 생성됩니다
- 계약 정산(type: contract)도 자동 생성됩니다
- 계약 정산 금액 우선순위:
  - 1순위: 파트너 개별 계약 단가(`partner_programs.contract_commission`)
  - 2순위: 캠페인 기본 계약 단가(`campaigns.contract_amount`)

**시나리오 3: 유효 처리 후 나중에 계약완료로 바꾸면**
- 유효 정산은 이미 생성되어 있으므로 중복 생성하지 않습니다
- 계약 정산만 추가로 1건 생성됩니다

**주의사항**:
- **파트너가 없는 리드는 정산이 생성되지 않습니다**: `partner_id`가 비어 있는 리드(파트너 추천 없이 직접 유입된 리드)는 정산을 건너뜁니다
- **정산은 자동 삭제되지 않습니다**: 실수로 유효 처리한 후 취소해도, 이미 생성된 정산은 자동으로 삭제되지 않습니다. 수동으로 확인하고 필요하면 Supabase에서 직접 삭제해야 합니다
- **정산 금액은 생성 시점에 확정됩니다**: 나중에 캠페인 단가를 바꿔도 이미 생성된 정산의 금액은 변하지 않습니다

### DB 트리거 관련

**자동 정산 생성 트리거 (`auto_create_settlement`)**:
- `contract_status='completed'` AND `is_valid=true` 일 때 자동 발동하여 정산 레코드를 생성합니다
- `partner_id`가 NULL인 리퍼럴은 정산 생성을 건너뜁니다 (2026-03-23 수정 완료)
- 유효 처리(is_valid=true)와 계약 완료(completed)를 동시에 변경하면 정산이 중복 생성될 수 있으므로 주의

**파트너 코드 매칭 트리거 (`match_partner_by_code`)**:
- `partners` 테이블에서 글로벌 코드를 조회합니다
- 앱에서 먼저 `partner_id`를 설정하면 이 트리거는 우회됩니다

### 추천코드 매핑 주의

- 추천코드는 `partner_programs.referral_code` 기준입니다 (광고주별로 고유)
- `partners` 테이블의 글로벌 코드가 아닙니다
- inquiry API와 webhook 모두 `partner_programs` 테이블에서 조회합니다

### 인증 방식 이중 구조

- **파트너**: Supabase Auth (이메일/비밀번호) — 업계 표준 보안
- **광고주**: 커스텀 세션 토큰 (`advertiser_sessions` 테이블) — Supabase Auth를 사용하지 않는 별도 방식
- 두 인증 체계가 완전히 분리되어 있으므로 운영 시 주의

### 멱등성 (중복 방지)

- Airtable 웹훅: `airtable_record_id` unique index로 같은 레코드 중복 처리 방지
- Airbridge 웹훅: `external_event_id`로 중복 이벤트 방지
- 문의 폼: 같은 광고주 + 같은 연락처로 90일 이내 중복 접수 차단

### 데이터 격리 (멀티테넌트)

- 모든 데이터(리드, 정산, 파트너 프로그램)는 `advertiser_id`로 격리됩니다
- 광고주 A의 데이터는 광고주 B에게 보이지 않습니다
- 파트너는 복수 광고주 프로그램에 동시 참가 가능하며, 프로그램마다 다른 추천코드를 받습니다

### 운영 시 자주 겪는 상황

**"파트너가 리드를 보냈는데 광고주 대시보드에 안 보여요"**
1. 파트너의 추천 코드가 유효한지 확인 (`partner_programs` 테이블)
2. 파트너 상태가 `approved`인지 확인
3. 문의 폼을 올바른 광고주 URL로 접근했는지 확인
4. `referrals` 테이블에서 직접 데이터 존재 여부 확인

**"계약 완료로 바꿨는데 정산이 안 생겼어요"**
1. `is_valid`가 `true`인지 확인 (false면 정산 안 됨)
2. `partner_id`가 NULL인지 확인 (NULL이면 정산 건너뜀)
3. Supabase SQL Editor에서 `settlements` 테이블 직접 확인

**"파트너가 비밀번호를 잊어버렸어요"**
- 파트너: `/reset-password`에서 이메일로 재설정 가능
- 광고주: `/advertiser/settings`에서 변경 가능. 잊어버린 경우 Supabase에서 직접 초기화

**"Airtable에서 리드가 들어오지 않아요"**
1. API Key가 올바른지 확인 (Settings에서 재확인)
2. Airtable Automation이 활성화 상태인지 확인
3. Webhook URL: `https://referio.kr/api/webhook/airtable`
4. 요청 헤더에 `X-API-Key`가 포함됐는지 확인
5. Vercel 로그에서 에러 메시지 확인

## 14. 광고주 유형별 운영 방식 비교

Referio는 광고주 유형(`advertiser_type`)에 따라 운영 방식이 완전히 다릅니다. 새 광고주를 온보딩할 때 반드시 유형을 먼저 확인하세요.

### 14.1 inquiry (문의 기반, 기본형) — 예: 한화비전 키퍼메이트

**대상**: B2B 서비스 광고주. 고객이 이름과 연락처를 남기는 방식

**흐름**:
```
파트너 → 추천 링크 공유 → 고객이 문의폼 작성 → 리드 등록
→ 광고주가 직접 전화/미팅 → 유효 처리 → 계약 완료 → 정산
```

**특징**:
- 고객 이름, 연락처가 DB에 저장됨
- 광고주가 수동으로 리드 상태를 변경 (pending → call_1 → call_2 → completed)
- 영업 퍼널 관리(CRM)가 핵심
- Airtable 연동 가능 (기존 CRM과 병행)
- 리퍼럴 관리 화면에서 칸반 보드로 영업 파이프라인 관리

**운영 포인트**:
- 리드 유입 후 빠른 초기 연락이 중요 (pending 상태 7일 초과 모니터링)
- 유효/무효 판단을 적시에 해야 파트너 신뢰 유지
- CRM 필드(영업 담당자, 우선순위, 라벨, 다음 할 일) 적극 활용 권장

### 14.2 event_tracking (이벤트 추적형) — 예: 밀리의서재

**대상**: B2C 앱 서비스 광고주. 앱 설치, 가입, 구독 등 디지털 이벤트를 추적하는 방식

**흐름**:
```
파트너 → 트래킹 링크 공유 → 고객이 앱 설치/가입/구독
→ 에어브릿지(MMP)가 이벤트를 자동 전송 → Referio가 자동 기록
→ 구독 이벤트 수신 시 → 자동 계약완료 → 정산 자동 생성
```

**특징**:
- 고객 이름/연락처가 없음 — device_id나 user_id로만 식별
- 사람이 수동으로 상태를 바꿀 필요 없음 — 이벤트 수신 시 자동 처리
- 퍼널: 설치(install) → 가입(sign_up) → 구독(subscribe)
- 광고주 대시보드에 "이벤트 현황" 전용 메뉴가 표시됨
- 에어브릿지 같은 MMP(모바일 측정 파트너) 연동 필수

**운영 포인트**:
- 에어브릿지 포스트백 설정이 정상 작동하는지 초기 모니터링 중요
- 퍼널 전환율(설치→가입→구독) 추이 확인
- 이상 트래픽(비정상적으로 높은 설치 수 등) 모니터링
- 리퍼럴 관리 화면보다 이벤트 현황 화면을 주로 사용

### 14.3 hybrid (혼합형)

**대상**: 문의 기반과 이벤트 추적을 모두 사용하는 광고주

**특징**:
- inquiry + event_tracking 기능이 모두 활성화됨
- 사이드바에 리퍼럴 관리와 이벤트 현황 메뉴가 모두 표시됨
- 현재 이 유형을 사용하는 실 고객은 없음 (시스템은 구현 완료)

### 14.4 유형별 운영 방식 요약 비교

| 항목 | inquiry | event_tracking | hybrid |
|------|---------|---------------|--------|
| 고객 식별 | 이름, 연락처 | device_id, user_id | 둘 다 |
| 리드 생성 | 문의폼/Airtable | MMP 자동 전송 | 둘 다 |
| 상태 변경 | 광고주 수동 | 자동 | 혼합 |
| 핵심 관리 화면 | 리퍼럴 관리 (칸반) | 이벤트 현황 | 둘 다 |
| CRM 필드 활용 | 높음 | 낮음 | 상황에 따라 |
| Airtable 연동 | 가능 | 불필요 | 선택적 |
| 실 사용 고객 | 한화비전 키퍼메이트 | 밀리의서재 | 없음 |

|----------|----------|------|

새 광고주를 Referio에 추가할 때 아래 순서로 진행합니다.

| 순서 | 작업 | 상세 |
|------|------|------|
| 1 | 광고주 레코드 생성 | Supabase → `advertisers` 테이블에 회사 정보 입력 |
| 2 | 로그인 정보 설정 | 광고주 ID(텍스트), 회사명, 로그인 사용자 ID, 비밀번호 설정 |
| 3 | 캠페인 생성 | 유효 단가, 계약 단가 설정. 중복 체크 기간, 유효 판정 기한 설정 |
| 4 | 브랜딩 설정 | 회사 로고 업로드, 테마 색상 설정 (문의폼에 반영됨) |
| 5 | 광고주 유형 설정 | `advertiser_type` 결정: inquiry(기본) / event_tracking / hybrid |
| 6 | Airtable 연동 (필요 시) | 광고주 설정 → Airtable 위저드 4단계 안내 |
| 7 | 파트너 가입 링크 공유 | `/signup/{광고주ID}` URL을 광고주에게 전달하여 파트너 모집에 활용 |
| 8 | 문의 폼 URL 확인 | `/inquiry/{광고주ID}` URL이 정상 동작하는지 테스트. 브랜딩 반영 확인 |
| 9 | 테스트 리드 발송 | 문의폼으로 테스트 리드 직접 제출 → 광고주 대시보드에 표시되는지 확인 |
| 10 | 이메일/Slack 알림 확인 | 테스트 리드 접수 시 이메일과 Slack 알림이 정상 발송되는지 확인 |

### 15.4 기존 파트너 이관 체크리스트

이미 다른 시스템에서 활동하던 파트너를 Referio로 옮길 때의 절차입니다 (한화비전 키퍼메이트 사례 참고).

| 순서 | 작업 | 상세 |
|------|------|------|
| 1 | 파트너 레코드 생성 | `partners` 테이블에 이름, 이메일, 채널 등 정보 입력 |
| 2 | Auth 계정 생성 | Supabase Auth에 이메일 기반 계정 생성 |
| 3 | auth_user_id 연결 | `partners.auth_user_id`에 Auth 계정의 UUID를 입력하여 연결 |
| 4 | 파트너 프로그램 등록 | `partner_programs` 레코드 생성 (status=approved, tier 설정) |

## 16. CRM 데이터 분석 가이드

현재 Referio 시스템에서 뽑을 수 있는 핵심 지표와 활용 방법입니다.

### 16.1 핵심 성과 지표 (KPI)

| 지표 | 계산 방법 | 의미 | 목표 기준 (참고) |
|------|----------|------|----------------|
| **유효율** | (유효 리드 수 / 전체 리드 수) x 100 | 들어온 리드 중 실제 잠재 고객 비율 | 60% 이상 양호 |
| **계약 전환율** | (계약 완료 수 / 유효 리드 수) x 100 | 유효 리드 중 실제 계약으로 이어지는 비율 | 20% 이상 양호 |
| **파트너당 평균 리드 수** | 전체 리드 수 / 활성 파트너 수 | 파트너 1명이 평균 몇 건의 리드를 유치하는지 | 월 3건 이상 |
| **파트너별 ROI** | (계약 수 x 계약 단가) / (누적 유효 단가 지급액) | 파트너에게 지급한 금액 대비 매출 기여도 | 1.0 이상이면 수익 |
| **리드 처리 속도** | 리드 유입일 → 유효/무효 결정일 평균 기간 | 리드를 얼마나 빨리 처리하는지 | 7일 이내 권장 |

### 16.2 분석별 확인 방법

**파트너 성과 비교**:
- 광고주 대시보드 → 파트너 관리에서 이달 리드 수, 계약 수 확인
- 리포트 탭에서 TOP 파트너 위젯 삽입
- CSV 내보내기 후 Excel에서 상세 분석

**영업 퍼널 전환율**:
- 리퍼럴 관리 상단의 퍼널 카드에서 각 단계별 리드 수 확인
- pending → call_1 → call_2 → call_3 → completed 각 단계별 감소율 파악
- 감소가 큰 구간이 병목(영업에서 개선이 필요한 지점)

**월별 리드 유입 트렌드**:
- CSV 내보내기 후 월별 그룹핑
- 또는 리포트 탭에서 리퍼럴 현황 위젯 활용

**정산 현황 분석**:
- 관리자 대시보드(`/admin`)에서 전체 정산 건수, 대기/완료/총액 확인
- CSV 내보내기로 월별 정산 추이 분석

### 16.3 데이터 추출 방법

현재 시스템에서 데이터를 추출하는 방법은 3가지입니다:

1. **CSV 내보내기** (가장 쉬움): 정산 관리 화면에서 "CSV 내보내기" 버튼 클릭. Excel에서 피벗 테이블 등으로 분석
2. **리포트 탭 위젯** (시각적): 광고주 리포트 탭에서 통계 위젯을 삽입하여 보고서 형태로 확인. PDF 다운로드 가능
3. **Supabase 직접 조회** (고급): Supabase Dashboard → SQL Editor에서 직접 쿼리. 복잡한 분석이 필요할 때 사용

## 17. 이메일 마케팅 전략

### 17.1 이메일 인프라

| 항목 | 값 |
|------|-----|
| 발송 서비스 | Resend (resend.com) |
| 발신 도메인 | `updates.puzl.co.kr` (DKIM, SPF 인증 완료) |
| 발신 주소 | `noreply@updates.puzl.co.kr` |
| 발신자 이름 | `Referio` |
| 무료 플랜 한도 | 월 3,000건 / 일 100건 |

### 17.2 현재 운영 중인 이메일 (3종)

| 코드 | 이메일명 | 트리거 | 수신자 | 상태 |
|------|---------|--------|--------|------|
| P1 | 파트너 가입 환영 | 파트너 최초 가입 시 | 신규 파트너 | 운영 중 |
| P2 | 파트너 승인 알림 | 광고주가 파트너 승인 시 | 승인된 파트너 | 운영 중 |
| A1 | 광고주 새 리드 알림 | 문의폼/Airtable로 리드 생성 시 | 광고주 담당자 | 운영 중 |

### 17.3 구현 예정 이메일 (5종, 우선순위 순)

| 코드 | 이메일명 | 트리거 | 비즈니스 가치 |
|------|---------|--------|-------------|
| P3 | 리드 유효 확정 알림 | 광고주가 리드를 유효 처리 시 | 파트너 동기부여 핵심 순간 — 추가 추천 활동 유도 |
| A2 | 신규 파트너 신청 알림 | 파트너가 프로그램 신청 시 | 승인 지연 방지 → 파트너 이탈 방지 |
| P4 | 계약 완료 알림 | 광고주가 계약 완료 처리 시 | 파트너 신뢰 구축 — 커미션이 쌓이고 있다는 체감 |
| P5 | 정산 처리 알림 | 광고주가 정산 완료 처리 시 | 신뢰 및 투명성 — 파트너 장기 유지의 핵심 |
| P6 | 비활성 리마인더 (월 1회) | 마지막 추천 후 30일 경과 | 휴면 파트너 재활성화 — 저비용 고효율 리텐션 |

### 17.4 이메일 트리거 맵

```
파트너 여정
--------------------------------------------
가입           → [P1] 환영 이메일 (운영 중)
프로그램 승인   → [P2] 승인 알림 (운영 중)
리드 유효      → [P3] 유효 확정 알림 (예정)
리드 계약      → [P4] 계약 완료 알림 (예정)
정산 처리      → [P5] 정산 알림 (예정)
30일 비활성    → [P6] 리마인더 (예정)

광고주 여정
--------------------------------------------
신규 리드 접수  → [A1] 리드 알림 (운영 중)
파트너 신청    → [A2] 파트너 신청 알림 (예정)
```

### 17.5 이메일 콘텐츠 가이드라인

- **1개 이메일 = 1개 핵심 메시지 + 1개 CTA 버튼**: 여러 내용을 한 이메일에 넣지 않습니다
- **수치 강조**: 금액, 건수, 달성률 등 구체적 숫자를 포함하면 오픈율이 높아집니다
- **제목 작성**: 구체적 수치 포함, 이모지 1개 허용 (끝에 배치)
- **발송 성공률 목표**: 99% 이상, 오픈율 40% 이상, 클릭률 15% 이상
- **Resend 대시보드에서 발송 성공률, 오픈율, 바운스율 모니터링 가능**

## 18. 앞으로의 방향 (Roadmap)

### 단기 (1~2주)

- 한화비전 키퍼메이트 파트너 96명 비밀번호 설정 이메일 일괄 발송

- 비밀번호 재설정 플로우 개선
- 웹훅 보안 강화 (HMAC 서명, Rate Limiting)

### 장기 (3개월+)

- 멀티 캠페인 지원 (광고주당 복수 캠페인)
- 글로벌 CRM 연동 (HubSpot, Salesforce)
- 파트너 소셜 로그인 (Google/Kakao)

- Referio 자체 추천 프로그램 (수익모델 확정 후)
- 연동 마켓플레이스 (외부 개발사 커넥터)

## 19. 자주 묻는 질문 (FAQ)

### 서비스 일반

**Q. Referio는 무료인가요?**
현재 공개 요금제는 없습니다. 광고주별 커스텀 계약으로 운영 중이며, 도입 문의는 별도 연락으로 진행합니다.

**Q. 파트너가 여러 광고주의 프로그램에 동시에 참가할 수 있나요?**
네. 파트너는 마켓플레이스에서 원하는 만큼 프로그램에 신청할 수 있고, 프로그램마다 별도 추천코드가 발급됩니다.

**Q. 추천 링크를 클릭한 고객이 추천코드를 직접 입력해야 하나요?**
아니요. 추천 링크에 코드가 이미 포함되어 있어서 고객은 링크만 클릭하면 됩니다. 이름과 연락처만 입력하면 자동으로 파트너에게 귀속됩니다.

### 광고주 관련

**Q. 정산은 어떻게 지급하나요?**
현재 Referio에서 직접 송금하는 기능은 없습니다. 광고주가 실제 계좌이체로 파트너에게 송금한 후, Referio 시스템에서 "완료 처리" 버튼을 눌러 기록합니다.

**Q. 캠페인 설정을 바꾸면 기존 리드에도 적용되나요?**
아니요. 변경 이후 새로 들어오는 리드부터 적용됩니다. 기존 리드의 정산 금액은 변경되지 않습니다.

**Q. 동일한 고객이 여러 파트너 링크를 통해 문의하면 어떻게 되나요?**
중복 체크 기능이 있습니다. 같은 광고주에 같은 연락처로 90일 이내 재문의 시 "이미 접수된 문의"로 차단됩니다.

**Q. 특정 파트너에게만 다른 단가를 적용할 수 있나요?**
네. `partner_programs`에서 해당 파트너의 `lead_commission`(유효 단가)과 `contract_commission`(계약 단가)을 설정하면, 캠페인 기본 단가 대신 이 값이 적용됩니다. 설정하지 않으면 캠페인 기본 단가가 적용됩니다.

### 파트너 관련

**Q. 추천 링크의 형식은 어떻게 되나요?**
`https://referio.kr/inquiry/{광고주ID}?ref={추천코드}` 형태입니다. 파트너 대시보드에서 복사 버튼을 눌러 바로 사용하면 됩니다.

**Q. 정산 금액은 어떻게 결정되나요?**
광고주가 캠페인에서 설정한 유효 단가(리드 유효 시)와 계약 단가(계약 완료 시)에 따라 자동 계산됩니다. 파트너별 개별 단가가 설정되어 있으면 그 값이 우선 적용되고, 없으면 캠페인 기본 단가가 적용됩니다. 티어가 높을수록 더 높은 단가가 적용될 수 있습니다.

**Q. 내가 유치한 고객의 영업 진행 상황을 볼 수 있나요?**
네. 파트너 대시보드의 "고객 목록" 탭에서 내가 유치한 리드의 계약 상태를 확인할 수 있습니다. 단, 상세 영업 내용(메모 등)은 광고주만 볼 수 있습니다.

### 운영/기술 관련

**Q. 데이터를 직접 수정해야 할 때 어떻게 하나요?**
Supabase Dashboard(https://supabase.com) → 해당 프로젝트 → Table Editor에서 직접 수정합니다. 삭제는 복구가 어려우므로 반드시 책임자 확인 후 진행하세요.

**Q. 광고주 인증과 파트너 인증이 왜 다른가요?**
광고주는 하나의 회사 계정에 여러 사용자가 접속하는 멀티유저 구조이고, 파트너는 개인 계정입니다. 설계 시 두 가지 다른 요구사항을 충족하기 위해 분리되었습니다.

**Q. 서비스가 다운되면 어떻게 확인하나요?**
Vercel Dashboard에서 배포 상태와 로그를 확인할 수 있습니다. Supabase Dashboard에서 DB 상태를 확인할 수 있습니다.

**Q. 유효 처리를 실수로 눌렀는데 정산이 생겼어요. 어떻게 취소하나요?**
자동 생성된 정산은 자동으로 삭제되지 않습니다. Supabase Dashboard → `settlements` 테이블에서 해당 정산 레코드를 직접 삭제하거나 상태를 `rejected`로 변경하세요. 그리고 리퍼럴의 `is_valid`를 `null`(미확인)로 되돌리세요.
